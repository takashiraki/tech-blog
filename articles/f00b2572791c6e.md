## `Redis`とは

まずは、公式に載っていた文章を引っ張ってきます。

> Redis is an open source data structure server. It belongs to the class of NoSQL databases known as key/value stores. Keys are unique identifiers, whose value can be one of the data types that Redis supports. These data types range from simple Strings, to Linked Lists, Sets and even Streams. Each data type has its own set of behaviours and commands associated with it.
>
> Redisはオープンソースのデータ構造サーバーです。キー/値ストアとして知られるNoSQLデータベースの一種に分類されます。キーは一意の識別子であり、その値はRedisがサポートするデータ型のいずれかを取ることができます。これらのデータ型は単純な文字列から、連結リスト、集合、さらにはストリームにまで及びます。各データ型には固有の動作と関連するコマンドが用意されています。

らしいです。まずはキーバリューでデータを保持していて、`NoSQL`の仲間らしいですね。さらに

> Redis is famous for being an extremely fast database. This speed comes from the fact that it stores and serves all data from memory rather than disk. Redis is durable, so your data will be persisted, but all reads will be from a copy of the data held in memory. This makes Redis an excellent choice for applications that require real time data access.
>
> Redisは極めて高速なデータベースとして知られています。この高速性は、すべてのデータをディスクではなくメモリから保存・提供するという特性に由来します。Redisは永続性を持つためデータは保持されますが、すべての読み取り操作はメモリ内に保持されたデータのコピーから行われます。この特性により、リアルタイムデータアクセスを必要とするアプリケーションに最適な選択肢となります。

らしく、めっちゃ早いインメモリデータベースらしいです。また、

> Redis is also often used as a cache, and has specific functionality to support this. Redis can be extended with new functionality using plugin modules. We'll see how to use some of these as we make our way through the course.
>
> Redisはキャッシュとしても頻繁に利用され、これをサポートする特定の機能を備えています。Redisはプラグインモジュールを用いて新機能を拡張できます。コースを進めていく中で、これらのいくつかを活用する方法を見ていきましょう。

まとめると、**『キーバリュー型のインメモリデータベースで、とっても早い。キャッシュとしても頻繁に利用される』** とのことらしいです。

### セッションストア
まずはセッションストアについてです。今回は

- Vue.js
- Laravel

これらを使ってログイン・ログアウトのみの簡単なアプリを作っています。

`sanctum`を使っているので、`Session driver`を`cookie`にしています。これでも問題なくログイン・ログアウトなどはできるのですが、一般的なブラウザの制約では

- 容量が4KBまで
- 20個までしか保存できない

という制約があるため、cookie制約に引っかかってしまうことがあるでしょう。この場合、LaravelではSession driverとして

- file - sessions are stored in storage/framework/sessions.
- cookie - sessions are stored in secure, encrypted cookies.
- database - sessions are stored in a relational database.
- memcached / redis - sessions are stored in one of these fast,
- cache-based stores.
- dynamodb - sessions are stored in AWS DynamoDB.
- array - sessions are stored in a PHP array and will not be persisted.

この辺が選べるっぽいのですが、fileだとそもそもサーバー間でセッション情報共有できないし、database使うほどでもないしなーというときにredisを使うことができます！

![Redis session storeの例①](/images/redis_session_store_1.png)

- ユーザーはフロントエンド（VueとかReactとか）から何かしらのリクエストをサーバーに向けて投げつける
- サーバー側はLaravelのソースコードが乗っかっている（認証機能付き）

というシンプルな構成だったとします。この時Laravelの.envが下記内容になっていたとします。

```text:.env
SESSION_DRIVER=cookie
```

ここで、リクエストが大量に飛んできて、スケールアウトしたとします。この時、`redis`にしていることで、スケールアウトした時もセッション情報が無事共有されます。そのため、認証エラーなどならずに済みます！
とっても便利ですね。

![Redis session storeの例②](/images/redis_session_store_2.png)

```text:.env
SESSION_DRIVER=redis
```

`Laravel`の`.env`を変更するだけで、これと言って特に意識をすることなくセッション情報をRedisで管理できるようになります。`Laravel`はすごいですね...

![Redis session storeの例③](/images/redis_session_store_3.png)

どのデータベースに保存されているからは、Laravelの`config/database.php`におそらく書いてあるかと思います。

```php:config/database.php
'redis' => [
    // ~~~省略~~~
    'default' => [
        'url' => env('REDIS_URL'),
        'host' => env('REDIS_HOST', '127.0.0.1'),
        'username' => env('REDIS_USERNAME'),
        'password' => env('REDIS_PASSWORD'),
        'port' => env('REDIS_PORT', '6379'),
        'database' => env('REDIS_DB', '0'),
        'max_retries' => env('REDIS_MAX_RETRIES', 3),
        'backoff_algorithm' => env('REDIS_BACKOFF_ALGORITHM', 'decorrelated_jitter'),
        'backoff_base' => env('REDIS_BACKOFF_BASE', 100),
        'backoff_cap' => env('REDIS_BACKOFF_CAP', 1000),
    ],
    // ~~~省略~~~
],
```

おそらくsession情報は0番に保存されているかと思います。`redis-cli`で確認をすると

```bash
redis:6379> SELECT 0
OK
redis:6379> SCAN 0
1) "0"
2) 1) "laravel-cache-43Vnv4LqtYDBpHneDKfmUkoSksCiM3BRzx3rhw2c"

redis:6379[1]> SELECT 1
OK
redis:6379[1]> SCAN 0
1) "0"
2) (empty array)
```

このように0番に保存されているのがわかるかと思います。

redis insightでも確認してみましょう。

![Redis session storeの例](/images/redis_insight_1st.png)

このように、ログイン前にはなにも保存されていませんが、ログイン後に`redis insight`を確認すると

![Redis session storeの例](/images/redis_insight_2nd.png)

### まずは自前で実装してみる

- Redis RPush / LPopを利用した実装

これら2つを用いて実装してみます。

#### Redis RPush / LPopを利用した実装

まずはRPushするソースコードです。

```php:app/Console/Commands/RedisPush.php
<?php

declare(strict_types=1);

namespace App\Console\Commands;

use Carbon\CarbonImmutable;
use Illuminate\Console\Command;
use Illuminate\Support\Facades\Redis;

class RedisPush extends Command
{
    /**
     * The name and signature of the console command.
     *
     * @var string
     */
    protected $signature = 'app:redis-push';

    /**
     * The console command description.
     *
     * @var string
     */
    protected $description = 'Command description';

    /**
     * Execute the console command.
     */
    public function handle()
    {
        for ($i = 0; $i < 500; $i++) {
            $timestamp = CarbonImmutable::parse(CarbonImmutable::now())->timezone('Asia/Tokyo')->format('YmdHis');
            $prefix = 'order' . $i . '_' . $timestamp . '_';
            $id = uniqid($prefix, true);
            Redis::rpush('order.queue', json_encode(['order_id' => $id]));
            $this->info("Pushed order ID: {$id} to Redis queue");
        }
    }
}
```

やっていることは、適当なID作って、RedisにRPushをしています。

次にRedisからPopする実装です。


```php:php:app/Console/Commands/RedisPop.php
<?php

declare(strict_types=1);

namespace App\Console\Commands;

use Illuminate\Console\Command;
use Illuminate\Support\Facades\Redis;

class RedisPop extends Command
{
    /**
     * The name and signature of the console command.
     *
     * @var string
     */
    protected $signature = 'app:redis-pop';

    /**
     * The console command description.
     *
     * @var string
     */
    protected $description = 'Command description';

    /**
     * Execute the console command.
     */
    public function handle()
    {
        $this->info('LPopRedisWorker started.');

        while (true) {
            $result = Redis::blpop('order.queue', 0);

            if (! $result) {
                continue;
            }

            $payload = json_decode($result[1], true);
            $orderId = $payload['order_id'] . PHP_EOL;

            // 実際の処理（例：メール送信・DB保存など）
            $this->process($orderId);

            $this->info("Handled order: {$orderId}");
        }
    }

    private function process(string $id): void
    {
        echo "Processing ID: {$id}";
    }
}
```

Pop側はひたすらRedis見に行って、対象のデータがPushされていないかを確認し、Pushされたらログに出すだけの処理をしています。

実際に下記コマンドをターミナルで開くと、おそらく確認できると思います。

```bash
# ターミナル1でPOPできるようにしておく
php artisan app:redis-pop

# その後ターミナル2でPUSHする
php artisan app:redis-push
```

こうすると、ターミナル1でおそらくログがたくさん出てくるかと思います。

#### Laravel Queue

続いて、Laravel queueを使って実装してみます。

https://laravel.com/docs/12.x/queues

まずはJobの作成をします。

```php:app/Jobs/JobCreated.php
<?php

declare(strict_types=1);

namespace App\Jobs;

use Illuminate\Contracts\Queue\ShouldQueue;
use Illuminate\Foundation\Queue\Queueable;
use Illuminate\Support\Facades\Log;

class JobCreated implements ShouldQueue
{
    use Queueable;

    /**
     * Create a new job instance.
     */
    public function __construct(
        public readonly string $jobId
    ) {
        //
    }

    /**
     * Execute the job.
     */
    public function handle(): void
    {
        Log::info('job created', ['jobId' => $this->jobId]);
    }
}
```

次に、JobをDispatchするコマンドの作成をします。

```php:app/Console/Commands/DispatchJob.php
<?php

declare(strict_types=1);

namespace App\Console\Commands;

use App\Jobs\JobCreated;
use Illuminate\Console\Command;
use Illuminate\Support\Str;

class DispatchJob extends Command
{
    /**
     * The name and signature of the console command.
     *
     * @var string
     */
    protected $signature = 'app:dispatch-job';

    /**
     * The console command description.
     *
     * @var string
     */
    protected $description = 'Command description';

    /**
     * Execute the console command.
     */
    public function handle()
    {
        for ($i = 0; $i < 100; $i++) {
            $uuid = Str::uuid()->toString();

            $job = new JobCreated($uuid);
            dispatch($job);
        }
    }
}
```

この状態でターミナルをふたつ開いて、下記コマンドを実行すると、おそらくqueueが動かせるかと思います。

```bash
# ターミナル1でまずはワーカーを動かす
php artisan queue:work

# 次にターミナル2でjobをdispatchする
php artisan app:dispatch-job
```

これでおそらくターミナル1にたくさんログが出てくるかと思います！！
